# Version 7 (v_latest)

**Status:** target specification — north star for reverse versioning (v7 → v1).

**Approach:** Define **v7 first** as the full system we want. Later versions
(v6, v5, … v1) are **subsets** of v7 — each removes or defers capability until
v1 is the minimal shippable core. Do not implement forward from v1 assuming
v7; **derive** v6..v1 by asking “what can we drop and still ship?”

**Reference scenario (coverage, not a cap):**
[`docs/scenarios/complex_two_source.md`](scenarios/complex_two_source.md) — two
publishers, a compose DAG, dual encoders, and multiple sinks. This is the
**minimum graph we must run in integration tests** to call v7 done. It does
**not** limit manifest size: v7 supports **any number** of sources, decoders,
composes, encoders, and sinks (dozens or hundreds), all wired the same way.

Earlier design notes (`docs/compose.md`, `docs/events.md`, …) may disagree on
details; **v7.md wins** for v7 scope. Other docs are not updated by this file.

---

## Scale (no artificial limits)

v7 manifest and runtime impose **no fixed maximum** on:

| Entity | v7 rule |
|--------|---------|
| **Sources** | Unbounded — each publisher is a `source_id` |
| **Decoders** | One per `(source, stream_type)` as needed; many sources ⇒ many decoders |
| **Composes** | Unbounded DAG — fan-in/fan-out via `inputs[]` |
| **Encoders** | Unbounded — each binds to compose (or other) media output |
| **Sinks** | Unbounded — remux per source, encode per encoder, event JSONL with many compose inputs |

**`complex_two_source`** uses 2 sources, 4 decoders, 5 composes, 2 encoders, 6
sinks to exercise **every v7 capability once**. A production manifest might
have 200 sources, 50 composes, and 30 sinks; validation and wiring rules are
identical. Limits are **operational** (CPU, memory, disk, network), not schema.

---

## v7 in one paragraph

**Full-capability** streaming graph: N publishers, remux/encode/event sinks,
decoders, a **fully abstract compose** layer with **inference** and **scene**
families, arbitrary compose DAGs, encoders, and **Argus watching** — each
compose publishes a **metric map** on a **millisecond evaluation schedule**;
when configured thresholds pass, it **pushes** `event/...` output that other
composes or sinks can read. The **complex_two_source** scenario (2 publishers)
is the **integration proof** for v7, not a size limit.

## Architectural pillars

| Pillar | v7 rule |
|--------|---------|
| **Compose** | Totally **abstract** — `compose_type` selects concrete plugin; runtime knows `ICompose`, not inference/scene internals |
| **Media path** | Extended **pull-based** — `ReadPacket`, demand-driven, no compose thread |
| **Event path** | **Push-based** — compose **pushes** event packets into its event output when thresholds fire; readers still **pull** via subscription/`ReadPacket` |
| **Metrics** | Each compose type **exports a map of named metrics**; manifest **events** bind thresholds to metric keys |
| **Time** | `evaluation_window_ms` and `evaluation_interval_ms` only — **no packet-count windows in v7** |
| **Sinks** | **Remux sink** for encoded A/V (`source_id` + `inputs[]`); **event remux sink** for `event/*` (see below) |
| **Wiring** | **Inputs only** — no upstream declares downstream |
| **Scale** | **Unbounded** entities in manifest — scenario uses 2 sources as test coverage only |

## Compose abstraction

### Runtime surface

```text
ICompose (public)
  ├── Prepare(params) / Close()
  ├── ReadPacket()              → media output (pull-based upstream fill)
  └── SubscribeEvents()         → IComposeEventSubscription*

IComposeEventSubscription (public — what downstream readers hold)
  └── ReadPacket()              → event output (pull read from push-filled ring)

compose_type registry           → factory per string type
```

**Not on `ICompose`:** metric evaluation, threshold checks, and pushing event
packets into the ring are **internal** — runtime scheduler + compose plugin
implementation. Downstream code never calls evaluate/push; it only
**subscribes** and **`ReadPacket`** on the subscription.

The runtime **never** branches on “is this inference or scene?” beyond
validation and metric/event schema declared by the type.

### Compose families (v7 must ship both)

**1. Inference composes** — run models or scoring over windowed inputs.

| `compose_type` | Role in scenario |
|----------------|------------------|
| `clip_prompt` | MobileCLIP-style image/text scoring (existing code baseline) |
| `reasoning` | Prompt + instruction voice → text/reasoning media + metrics/events |

**2. Scene composes** — layout, transform, overlay on raw video (and optional audio).

| `compose_type` | Role in scenario |
|----------------|------------------|
| `video_change` | Apply text/reasoning directive to decoded A/V |
| `watermark` | Fuse location metadata onto video |
| `side_by_side` | Multi-panel layout (studio processed + guest) |

v7 **done** requires all six types **working** (not stubs): real inference where
applicable, real scene output for the scenario graph.

### Per-type contract

Each registered `compose_type` declares (static metadata used at manifest validate time and **info API**):

```text
ComposeTypeDescriptor
  compose_type              string id e.g. "reasoning", "watermark"
  family                    inference | scene
  allowed_input_kinds[]     source | decoder | compose
  allowed_output_packet_types[]
  allowed_event_packet_types[]    emit targets manifest may use
  metrics[]                 FIXED keys defined by compose author:
                              name, value_type (number | int | string | bool), description
  param_schema              type-specific manifest fields
```

**Metric keys are fixed per type** — the compose plugin author registers the
exact set (e.g. `reasoning` always exposes `confidence`, `directive_stability`,
`token_count`). Manifest `events[].metric` **must** match a registered key and
**`op` must be valid for that metric's `value_type`**. Otherwise **`upsert_manifest`
rejects the entire request** (same strict validation as unknown sources).

Example descriptor excerpt:

```json
{
  "compose_type": "reasoning",
  "family": "inference",
  "metrics": [
    { "name": "confidence", "value_type": "number", "description": "Model confidence 0..1" },
    { "name": "directive_stability", "value_type": "number", "description": "Stability across window" },
    { "name": "token_count", "value_type": "int", "description": "Tokens in last directive" },
    { "name": "last_error", "value_type": "string", "description": "Empty if ok" }
  ],
  "allowed_event_packet_types": [
    "event/reasoning_complete",
    "event/reasoning_failed",
    "event/reasoning_low_confidence"
  ]
}
```

**`reasoning` is one compose type** — the scenario uses two instances
(`reasoning-a`, `reasoning-b`) with different inputs/params; no separate
`reasoning_refine` type in v7.

**`watermark` is an explicit compose type** in v7 (not a generic overlay param).

## Metrics map and events

### Metric map (compose-owned)

On each **evaluation tick** (every `evaluation_interval_ms`), the compose
implementation:

1. Collects packets whose **presentation time** falls in the last
   `evaluation_window_ms` (wall-clock window on PTS/ms timeline).
2. Runs type-specific logic (model, sensor read, math, …).
3. Returns **`std::map<std::string, MetricValue>`** (or equivalent) — the
   **exported metrics** for this compose instance.

Example keys:

| Compose | Example metric keys |
|---------|---------------------|
| `clip_prompt` | `cosine_similarity`, `inference_latency_ms` |
| `reasoning` | `confidence`, `directive_stability`, `token_count` |
| `video_change` | `frames_applied`, `reject_count` |
| `watermark` | `location_age_ms`, `overlay_applied` |
| `side_by_side` | `panel_stale_count`, `layout_id` |

Metric names are **stable per compose_type**; manifest references them in
`events[]`.

### Event bindings (manifest)

Events are **not** hard-coded only inside C++. Manifest declares which metrics
trigger which `event/...` packets:

```json
{
  "compose_id": "reasoning-b",
  "compose_type": "reasoning",
  "evaluation_window_ms": 3000,
  "evaluation_interval_ms": 500,
  "inputs": [ ... ],
  "events": [
    {
      "metric": "confidence",
      "op": "lt",
      "value": 0.7,
      "emit": "event/reasoning_low_confidence",
      "severity": "warn"
    },
    {
      "metric": "directive_stability",
      "op": "gte",
      "value": 0.9,
      "emit": "event/reasoning_complete",
      "severity": "info"
    },
    {
      "metric": "last_error",
      "op": "eq",
      "value": "timeout",
      "emit": "event/reasoning_failed",
      "severity": "error"
    }
  ]
}
```

Supported **`op`** by **`value_type`**:

| `value_type` | Allowed `op` |
|--------------|--------------|
| `number`, `int` | `lt`, `lte`, `gt`, `gte`, `eq`, `neq` |
| `string` | `eq`, `neq` |
| `bool` | `eq`, `neq` |

Manifest **`value`** type must match metric type (number, string, or boolean).
Invalid combination → reject upsert.

**Re-emit (locked):** if a threshold stays true on consecutive evaluation ticks,
**emit again every tick** (every `evaluation_interval_ms`). No hysteresis in v7.

On threshold pass, compose **pushes** an event packet:

```json
{
  "event": "reasoning_complete",
  "compose_id": "reasoning-b",
  "metric": "directive_stability",
  "value": 0.94,
  "threshold": 0.9,
  "op": "gte",
  "evaluation_window_ms": 3000,
  "evaluation_interval_ms": 500,
  "window_start_ms": 118000,
  "window_end_ms": 121000,
  "severity": "info"
}
```

Multiple `events[]` rules may fire on the same tick. Multiple metrics may map
to different events independently.

### Push-based events vs pull-based media

| Output | Fill model | Reader |
|--------|------------|--------|
| **Media** (`video/raw`, `text/reasoning`, …) | Pull — downstream `ReadPacket` fills upstream | Decoder, compose, encoder |
| **Events** (`event/...`) | **Push** (internal) — runtime tick fills event ring | Subscriber **`ReadPacket`** on event subscription |

Downstream binds via **`SubscribeEvents()`**, then reads with
**`subscription->ReadPacket()`** — same fan-out pattern as media subscriptions.
Evaluation and ring push are **not** public `ICompose` methods.

Media `ReadPacket` must **not** block on event evaluation.

### Evaluation scheduler (locked)

- Ticks are driven by **presentation timeline (PTS in milliseconds)**, not
  packet count and not a detached wall-clock when media is flowing.
- Each compose instance sets **`evaluation_interval_ms`** in manifest (e.g.
  **`100`** → evaluate every 100 ms of presentation time advancement).
- **`evaluation_window_ms`** is the look-back width on that same timeline.
- When **no packets** arrive for a compose input, presentation time may stall;
  evaluation ticks **do not run ahead of timeline** (no synthetic ticks while idle).
- If a window contains **no usable input**, internal evaluation may produce an
  empty or partial metric map; threshold rules against missing metrics **do not fire**.

Implementation note: runtime advances compose clocks from **ingest PTS** (source
timeline) or from **output PTS** of upstream media pulls — exact binding is
implementation detail; manifest semantics are **presentation ms only**.

---

## Evaluation timing (milliseconds only)

| Field | Type | Meaning |
|-------|------|---------|
| `evaluation_window_ms` | integer | Width of look-back window on presentation timeline |
| `evaluation_interval_ms` | integer | Minimum time between evaluation ticks |

Rules:

- Window is **`[now_ms - evaluation_window_ms, now_ms]`** in presentation time
  (session-normalized PTS ms).
- A tick runs at most once per `evaluation_interval_ms` wall/presentation step.
- Packets in window may be **any** input stream the compose consumed (video,
  voice, text, location) — compose type defines how they enter the metric map.

No `evaluation_window` packet-count field in v7 manifest.

## Sinks (v7)

### 1. Remux sink (encoded media)

One sink per recorded program (or more if desired). Muxes **encoded** streams
from **`source_id`**:

```json
{
  "sink_id": "rec-a",
  "source_id": "live/studio",
  "output_root": "./record/studio",
  "mode": "record",
  "inputs": [
    { "packet_type": "video/h264", "stream_id": "video/main" },
    { "packet_type": "voice/aac", "stream_id": "voice/main" }
  ]
}
```

Pull-based from source subscription; interleave/mux on write (HLS).

### 2. Event remux sink

Dedicated sink for **event packets** — same remux/write machinery, different
input binding. Reads **`event/*`** from one or more compose event subscriptions
(v7: list compose ids or a single “event bus” compose — **easiest: per-compose
subscription list**):

**Event remux sink (locked):** `inputs[]` with `kind: compose`; output **JSONL**
(one JSON object per line per event packet) under `output_root`. No HLS for
events in v7.

```json
{
  "sink_id": "events-log",
  "sink_kind": "event_jsonl",
  "output_root": "./events",
  "inputs": [
    { "kind": "compose", "id": "reasoning-b", "packet_type": "event/*" },
    { "kind": "compose", "id": "watermark-a", "packet_type": "event/*" }
  ]
}
```

Encoded remux sinks keep existing HLS remux behavior (`sink_kind` optional /
default `remux_hls`).

### 3. Encode-branch sinks

Unchanged from scenario — bind to **`encoder_id`**, pull encoded packets.

## Reference graph (complex_two_source @ v7)

**Integration coverage only** — proves v7 end-to-end with two publishers. Same
rules apply to graphs of any size.

```text
Sources: live/studio (5 packet types), live/guest (2)

Remux (encoded):
  rec-a ← studio video/h264 + voice/aac
  rec-b ← guest video/h264 + voice/opus

Decoders (4):
  video-dec-a, voice-dec-a, video-dec-b, voice-dec-b

Compose DAG (inputs only):
  reasoning-a   ← source prompt + voice/instruction (decoder)
  reasoning-b   ← reasoning-a (text/reasoning)
  video-change-a ← video-dec-a + voice-dec-a (main) + reasoning-b (text/reasoning)
  watermark-a   ← video-change-a + source location
  layout-a      ← watermark-a + video-dec-b + voice-dec-b (side_by_side)

Encoders:
  enc-live, enc-record ← layout-a (video/raw)

Encode sinks:
  sink-live ← enc-live
  sink-rec-a, sink-rec-b ← enc-record

Event sink:
  events-log ← reasoning-b, video-change-a, … (event/*)

Events: push on metric thresholds (ms windows) from inference + scene composes
```

### Data vs event for reasoning → video_change

**v7 choice (easier):** `video-change-a` reads **`text/reasoning` media**
from `reasoning-b` for processing. **`event/reasoning_complete`** is for
observers (event sink, automation, optional future composes) — not required
as the sole input to video_change.

## Info API (v7)

Runtime exposes compose type catalog for operators and manifest authors.

```text
GET /compose_types
GET /compose_types/{compose_type}
```

Response includes everything in **`ComposeTypeDescriptor`**: family, metrics
(fixed keys + `value_type`), allowed inputs/outputs/events, param schema,
human-readable description.

Does **not** mutate graph state. Used to build valid manifests offline and to
validate client-side before `POST /upsert_manifest`.

If manifest references unknown `compose_type`, unknown metric, invalid `op` for
metric type, or unknown param key → **400**, entire upsert rejected, graph unchanged.

### Example compose block (v7 complete)

```json
{
  "compose_id": "reasoning-a",
  "compose_type": "reasoning",
  "evaluation_window_ms": 5000,
  "evaluation_interval_ms": 1000,
  "inputs": [
    {
      "kind": "source",
      "id": "live/studio",
      "stream_id": "text/prompt",
      "packet_type": "text/prompt"
    },
    {
      "kind": "decoder",
      "id": "voice-dec-a",
      "stream_id": "voice/instruction",
      "packet_type": "voice/pcm"
    }
  ],
  "model_root": "./models/reasoning",
  "events": [
    {
      "metric": "confidence",
      "op": "lt",
      "value": 0.5,
      "emit": "event/reasoning_failed",
      "severity": "error"

}
```

Type-specific params (`model_root`, layout canvas, …) live beside `events[]`;
unknown keys rejected per `ComposeTypeDescriptor`.

## C++ plugin shape (illustrative)

### Public interfaces (subscribers see only this)

```cpp
class IComposeEventSubscription {
 public:
  virtual ReadResult ReadPacket() = 0;
};

class ICompose {
 public:
  virtual bool Prepare(const ComposeParams& params) = 0;
  virtual void Close() = 0;

  // Media (pull)
  virtual ReadResult ReadPacket() = 0;

  // Events — subscribe once, read many (fan-out via cursors)
  virtual IComposeEventSubscription* SubscribeEvents() = 0;
};
```

Event sink, compose inputs, and other readers:

```cpp
auto* sub = compose->SubscribeEvents();
while (auto pkt = sub->ReadPacket()) { /* handle event packet */ }
```

### Internal (not exposed on ICompose)

Metric collection, threshold comparison against manifest `events[]`, and
pushing into the event ring happen inside the **runtime + compose plugin** —
not callable by downstream graph nodes.

```cpp
// Illustrative — plugin-internal or runtime↔plugin private contract
struct MetricValue { /* number | int | string | bool */ };
using MetricMap = std::map<std::string, MetricValue>;

// Concrete compose may implement privately; runtime invokes on ms tick:
//   metrics = plugin->CollectMetrics(now_ms, window_start_ms);
//   runtime compares metrics to manifest EventRules;
//   runtime pushes matching event packets into event ring;
```

Runtime owns: ms scheduler, window packet cache by PTS, threshold comparison,
event ring push, subscription cursors. Compose plugin owns: type-specific metric
logic inside `CollectMetrics` (name illustrative, not public API).

| # | Decision |

| 1 | Event sink output = **JSONL** |

## Summary

v7 is **v_latest**: unbounded manifest graphs, abstract composes with
**inference** and **scene** plugins, **metric map → manifest-defined events**,
**ms windows**, **push events** with **pull read**, **remux sinks** for encoded
media and **JSONL** events. **`complex_two_source`** is the required integration
scenario — not a maximum size.
